Backend Implementation Plan

YouTube Playlist Viewer — 5 phases, each building on the last

Context: The frontend is built (React + TypeScript, 3 pages, mock data) but the backend is a Django + DRF scaffold with zero models and one hello-world endpoint. The app is now multi-user (per-user accounts, per-user YouTube connections, per-user notes) instead of single-user.

Phase 1 User Authentication

Users can register, login, logout. Protected endpoints reject unauthenticated requests.

Django concepts to learn

django.contrib.auth, DRF TokenAuthentication, permissions.IsAuthenticated, APIView

Endpoints

MethodPathPurpose
POST/api/auth/register/Create account (username, email, password, password2) → returns token
POST/api/auth/login/Login (username, password) → returns token (use DRF's obtain_auth_token)
POST/api/auth/logout/Deletes the request's auth token
GET/api/auth/me/Returns current user info

Steps

  1. Add rest_framework.authtoken to INSTALLED_APPS in settings
  2. Configure DEFAULT_AUTHENTICATION_CLASSES and DEFAULT_PERMISSION_CLASSES in REST_FRAMEWORK dict
  3. Create backend/api/serializers.py with RegisterSerializer
  4. Add register, login, logout, me views to views.py
  5. Wire URLs in urls.py
  6. Run python manage.py migrate

Files: settings.py, serializers.py (new), views.py, urls.py
New deps: None (DRF includes authtoken)

Phase 2 YouTube OAuth & API Client

Users can connect their YouTube account. Backend can fetch playlists/videos from YouTube Data API v3.

Django concepts to learn

OneToOneField, TextField, custom model methods, os.environ for config, OAuth 2.0 flow

YouTubeCredential Model

FieldTypeNotes
userOneToOneField → UserOne credential set per user
token_jsonTextFieldStores Credentials.to_json() from Google's library

Add helpers: get_credentials() → Credentials object, set_credentials(creds) stores JSON.

New Module: backend/api/youtube.py

Functions: build_flow(), get_auth_url(state), exchange_code(code), get_authenticated_service(credentials), fetch_playlists(service), fetch_playlist_items(service, playlist_id)

Endpoints

MethodPathPurpose
GET/api/youtube/auth-url/Returns {"url": "..."} — frontend redirects user here
GET/api/youtube/callback/Handles ?code=&state= from Google, stores credentials
GET/api/youtube/playlists/Lists user's YouTube playlists live (not yet stored in DB)

Key trick: Pass the DRF token inside the OAuth state parameter (as JSON) so you know which user is returning from Google.

Files: models.py, youtube.py (new), views.py, urls.py, settings.py, requirements.txt
New deps: google-api-python-client, google-auth-oauthlib

Endpoints (via DRF Router)

MethodPathPurpose
GET/api/playlists/List current user's playlists
POST/api/playlists/Create a playlist
GET/PATCH/DELETE/api/playlists/{id}/Retrieve/update/delete
POST/api/playlists/{id}/sync/Custom action: re-fetch from YT, mark removed videos
GET/api/videos/List videos (filterable by ?playlist=)
GET/api/videos/{id}/Video detail
GET/POST/api/notes/List/create notes
GET/PATCH/DELETE/api/notes/{id}/Retrieve/update/delete note

Sync Logic (video retention)

POST /api/playlists/{id}/sync/

1. Fetch current playlist items from YouTube API
2. DB videos whose youtube_id is NOT in API response → set removed_at = now()
3. Videos in API response but not in DB → create
4. Videos in both → update metadata (title, viewCount, etc.)
5. Update playlist.last_synced_at

Files: models.py, serializers.py, views.py, urls.py, admin.py
New deps: None

Phase 4 Gemini AI Integration

AI-powered video analysis and chat about videos.

Django concepts to learn

Calling external APIs from Django, serializer validation, environment variable management

New Module: backend/api/gemini.py

Functions for calling Gemini API with video context, formatting prompts.

Endpoints

MethodPathPurpose
POST/api/videos/{id}/analyze/Returns AI analysis (summary, key topics)
POST/api/videos/{id}/chat/Accepts {"message": "..."}, optionally {"history": [...]}, returns AI reply

Chat state: Start stateless — the frontend sends full conversation history each request. Add a ChatMessage model later if needed.

Files: gemini.py (new), views.py, urls.py, settings.py, requirements.txt
New deps: google-genai

Phase 5 Frontend Wiring (High-Level Reference)

Frontend consumes the new backend APIs instead of mock data and localStorage.

  1. Create frontend/src/services/api.ts — fetch wrapper with token management
  2. Add Login/Register pages with auth context
  3. Replace mock data imports with API calls on all pages
  4. Replace localStorage notes with API-backed notes
  5. Add YouTube connect button flow
  6. Adjust TypeScript types to match API response shapes

Implementation Order

Phase 1 (Auth) → Phase 2 (YouTube OAuth) → Phase 3 (Models + API) → Phase 4 (Gemini) → Phase 5 (Frontend)

Phases 2 and 3 are semi-parallel — you can build Phase 3 models first, then wire the sync action after Phase 2.

Verification

Test each phase with curl or Django's browsable API:

Key Design Decisions

DecisionChoiceWhy
AuthDRF TokenAuthenticationSimplest. JWT can come later.
User modelBuilt-in UserEnough for now. Don't over-engineer.
OAuth tokensSingle TextField with Credentials.to_json()Google's library handles serialization.
OAuth callback authstate param carries DRF tokenAvoids cookie complexity.
ViewsModelViewSet + DefaultRouterStandard DRF, minimal boilerplate.
Video removalremoved_at soft-deleteMatches requirement exactly.
Notesunique_together(user, video)One note per user per video.
Gemini chatStateless (history in request)Simpler backend. Add persistence later.
