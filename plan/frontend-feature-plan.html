Frontend Feature Plan

YouTube Playlist Viewer - companion plan for the Django + DRF backend implementation

Context: The current React + TypeScript frontend has three mock-data pages: playlist browser, playlist detail, and watch page with local Markdown notes. This plan describes the frontend features needed once the backend adds authentication, YouTube OAuth, stored playlists/videos, per-user notes, sync, removed-video tracking, and Gemini-powered analysis/chat.

1 Auth Shell

Make the app multi-user from the first screen: register, login, logout, current user, and protected routes.

UI to build
  • Login and register pages that call /api/auth/login/ and /api/auth/register/.
  • Auth context/provider that stores the DRF token and current user from /api/auth/me/.
  • Protected route wrapper for playlist, detail, and watch pages.
  • Header user area with username/email and logout action.
Acceptance checks
  • Logged-out users are redirected to login when opening protected pages.
  • Register and login both land on the playlist browser with a valid token.
  • Logout clears the token and prevents access to protected API calls.

2 YouTube Connection & Onboarding

Guide each user through connecting their own YouTube account before playlist data is available.

UI to build
  • Playlist-page empty state with a clear Connect YouTube action.
  • OAuth starter that calls /api/youtube/auth-url/ and redirects to the returned URL.
  • Return/callback handling state for success, cancellation, and connection failure.
  • Connected/disconnected status indicator in the header or account menu.
Acceptance checks
  • Users without credentials see onboarding instead of an empty playlist grid.
  • Clicking connect starts the backend-provided OAuth flow.
  • Connection errors are shown with a retry action.

3 API-Backed Playlist & Video Browsing

Replace mock data with backend data while preserving the existing three-page navigation model.

UI to build
  • Shared api.ts fetch wrapper with base URL, token header, JSON parsing, and unauthorized handling.
  • Playlist browser using GET /api/playlists/.
  • Playlist detail using GET /api/videos/?playlist={id}.
  • Watch page using GET /api/videos/{id}/ for title, metadata, and YouTube embed id.
  • TypeScript response types aligned with backend serializer fields.
Acceptance checks
  • Mock data imports are removed from page-level data flow.
  • Loading, empty, and error views appear on every API-backed page.
  • Unauthorized API responses return users to login.

4 Sync & Removed-Video UI

Expose playlist sync and make removed YouTube videos visible instead of silently disappearing.

UI to build
  • Manual sync button on playlist cards or playlist detail pages using POST /api/playlists/{id}/sync/.
  • Show last_synced_at in playlist metadata.
  • Sync loading, success, and failure feedback.
  • Removed-video badge and dimmed styling when removed_at is present.
  • Video filters: All, Available, and Removed.
Acceptance checks
  • Sync cannot be double-submitted while already running.
  • Removed videos show "Removed on {date}" and remain accessible for notes/history.
  • Filters update the video list without losing playlist context.

5 Per-Video Backend Notes

Replace global localStorage notes with one saved Markdown note per user per video.

UI to build
  • Load the current user's note for the selected video from /api/notes/.
  • Create or patch notes when content changes.
  • Keep the existing edit/preview Markdown interface.
  • Show save states: saving, saved, failed, and retry.
Acceptance checks
  • Notes are scoped to the active video and user.
  • Refreshing the watch page restores the saved note from the backend.
  • Network failures do not erase unsaved text.

6 AI Analysis & Chat Panel

Add Gemini-powered video analysis and stateless chat alongside notes on the watch page.

UI to build
  • Tabs on the watch sidebar: Notes and AI.
  • Analyze video action calling POST /api/videos/{id}/analyze/.
  • Summary/key topics result display.
  • Chat composer that sends message and full local history to POST /api/videos/{id}/chat/.
  • AI response pending, error, and retry states.
Acceptance checks
  • Users can switch between notes and AI without losing unsaved note text.
  • Chat history is included in each request during the active browser session.
  • AI failures are visible and do not break video playback.

7 Search & Filter Polish

Make large playlist libraries and long playlists manageable.

UI to build
  • Playlist search by title and channel.
  • Video search inside a playlist.
  • Sort videos by playlist position, published date, and view count where available.
  • Clear empty results state with a reset filter action.
Acceptance checks
  • Search and filters are keyboard-friendly.
  • Empty filtered results are distinct from a truly empty playlist.
  • Filter state does not corrupt navigation to the watch page.

8 Account & Settings Page

Give users a small control surface for account and integration status.

UI to build
  • Account page or menu showing username, email, and YouTube connection status.
  • Reconnect YouTube action using the existing OAuth flow.
  • Logout action and backend connectivity/error status.
  • Placeholder for future preferences such as default video filters or AI behavior.
Acceptance checks
  • Users can identify which account is active.
  • Reconnect flow is available without needing to log out.
  • Settings remains useful even before advanced preferences exist.

UI States to Implement

LoadingInitial page/API fetches and sync requests.
EmptyNo playlists, no videos, no notes, or no AI messages yet.
ErrorFailed API calls with retry affordances.
UnauthorizedExpired or missing token redirects to login.
SavingNotes are being persisted to the backend.
SyncedPlaylist has a visible last synced timestamp.
DisconnectedYouTube OAuth has not been completed or needs reconnect.
Removed VideoVideo is archived with a removed date but remains viewable as history.
AI PendingAnalysis or chat response is being generated.

Recommended Implementation Order

Auth shell → YouTube onboarding → API-backed browsing → Sync and removed videos → Backend notes → AI panel → Search/filter polish → Account/settings

This order follows the backend phases and avoids building UI that has no stable API surface yet.

Implementation Notes

AreaDefault approachReason
RoutingKeep the existing playlist, detail, and watch routes; add auth routes and protected wrappers.Preserves the current app shape while adding multi-user behavior.
Token storageStore the DRF token in browser storage through a single auth context.Matches the backend's simple TokenAuthentication design.
NotesSave one note per user/video and preserve local draft text during failures.Prevents data loss and matches the backend uniqueness rule.
